# RBG Planner

SLA-based autoscaler for [RoleBasedGroup](https://github.com/rolebasedgroup) Prefill/Decode disaggregated inference workloads.

The RBG Planner automatically scales Prefill and Decode roles within a RoleBasedGroup to meet TTFT (Time to First Token) and ITL (Inter-Token Latency) SLA targets using performance profiling data, load prediction, and real-time correction factors.

## Acknowledgements

The core planning algorithm, SLA profiling methodology, and performance interpolation approach in this project are derived from [NVIDIA Dynamo](https://github.com/ai-dynamo/dynamo). We are deeply grateful to the Dynamo team for their pioneering work on SLA-driven autoscaling for disaggregated inference. The Dynamo planner's design — combining offline profiling with online prediction and correction factors — forms the foundation of RBG Planner. This project adapts and re-implements those ideas to work natively with the Kubernetes RoleBasedGroup API, removing Dynamo runtime dependencies (etcd, NATS, DistributedRuntime) and making the approach available as a standalone, lightweight component for any RBG-based deployment.

### Component Roles

| Component | Description |
|-----------|-------------|
| **RBG Planner (Dynamo Planner Engine)** | The core autoscaler. Uses the Dynamo planner algorithm to observe metrics, predict load, compute replica requirements from profiling data, and apply scaling decisions. |
| **Engine Runtime (Patio Sidecar)** | A metrics abstraction layer deployed alongside inference engines. Provides unified `patio:*` Prometheus metrics regardless of the underlying engine (SGLang, vLLM, Dynamo Engine, etc.), making the planner engine-agnostic. |
| **Dynamo Global Planner** | An optional higher-level orchestrator that coordinates scaling decisions across multiple RBG instances. Manages global GPU budget allocation and multi-model scaling priorities. |
| **Profiling ConfigMap** | Performance profiling data that maps workload characteristics (ISL, concurrency, context length) to hardware metrics (TTFT, ITL, throughput/GPU). Generated via `inference-ext-cli` (with built-in SLA profiling pipeline) or AIC (AI Center profiling service). |
| **RBG Controller** | The Kubernetes controller that reconciles RoleBasedGroup specs into actual StatefulSets/Deployments. Receives scaling decisions from the planner via RBGSA patch. |

### Metric Sources

The planner supports multiple metric sources via the `METRIC_SOURCE` configuration:

| Source | Description |
|--------|-------------|
| `sglang` | Native SGLang Prometheus metrics (default) |
| `vllm` | Native vLLM Prometheus metrics |
| `patio` | Unified metrics from the Patio EngineRuntime sidecar (engine-agnostic, recommended for production) |

When using `patio`, the planner works identically with any inference engine — no reconfiguration needed when switching between SGLang, vLLM, or Dynamo Engine.

## How It Works

Each planning cycle (powered by the Dynamo planner algorithm):

1. **Observe** - Queries Prometheus for TTFT, ITL, request rate, ISL/OSL metrics (via Engine Runtime or native engine metrics)
2. **Predict** - Uses configurable load predictor (ARIMA/Constant/Prophet) to anticipate upcoming demand
3. **Compute** - Interpolates profiling data to determine required prefill/decode replicas that satisfy SLA targets
4. **Scale** - Applies scaling decisions via RBGSA scale subresource (with fallback to direct RBG patch)

Correction factors continuously adjust predictions based on observed vs. expected performance, compensating for real-world deviations from profiled behavior.

## Quick Start

This guide walks through the full workflow: generate profiling data, create a planner-integrated RBG deployment, apply it to the cluster, and observe the planner behavior.

### Prerequisites

- Kubernetes cluster with [RBG controller](https://github.com/rolebasedgroup/rbg) installed
- `kubectl` configured to access the cluster
- `inference-ext-cli` installed (from [rbg](https://github.com/rolebasedgroup/rbg))
- Prometheus monitoring stack deployed

### Step 1: Prepare Profiling Data

Option A — **Run SLA profiling via inference-ext-cli** (automated, recommended):

```bash
inference-ext-cli profile \
  --engine sglang \
  --model "Qwen/Qwen3-0.6B" \
  --engine-image "lmsysorg/sglang:latest" \
  --namespace profiling \
  --min-gpus 1 --max-gpus 4 \
  --ttft-sla 200 --itl-sla 20 \
  --output-dir ./profiling-results
```

This deploys temporary RBG instances, runs benchmarks at various GPU configurations, selects optimal parallelization, performs interpolation sweeps, and produces `prefill_raw_data.json` + `decode_raw_data.json`.

Option B — **Use existing profiling data** from AIC or prior runs:

```bash
ls ./profiling-results/
# prefill_raw_data.json
# decode_raw_data.json
```

### Step 2: Generate Deployment YAML

Use `inference-ext-cli generate` to produce a complete deployment with the planner role injected:

```bash
inference-ext-cli generate \
  --rbg-yaml ./my-sglang-pd.yaml \
  --enable-planner \
  --planner-image ghcr.io/rolebasedgroup/rbg-planner:latest \
  --model-name "Qwen/Qwen3-0.6B" \
  --profiling-source json \
  --prefill-json ./profiling-results/prefill_raw_data.json \
  --decode-json ./profiling-results/decode_raw_data.json \
  --ttft-sla 200 \
  --itl-sla 20 \
  --max-gpu-budget 8 \
  -o ./output/
```

Output:
```
output/
├── rbg.yaml                  # RBG with prefill + decode + planner roles
└── profiling-configmap.yaml  # ConfigMap with profiling data
```

### Step 3: Deploy to Cluster

```bash
# Apply profiling ConfigMap
kubectl apply -f ./output/profiling-configmap.yaml

# Apply the RBG (includes planner role)
kubectl apply -f ./output/rbg.yaml
```

Verify all roles are running:

```bash
kubectl get pods -l rbg.workloads.x-k8s.io/group-name=sglang-pd-inference
```

Expected output:
```
NAME                            READY   STATUS    AGE
sglang-pd-inference-prefill-0   1/1     Running   2m
sglang-pd-inference-decode-0    1/1     Running   2m
sglang-pd-inference-planner-0   1/1     Running   2m
```

### Step 4: Observe Planner Behavior

Watch the planner logs:

```bash
kubectl logs -l rbg.workloads.x-k8s.io/role-name=planner -f
```

Expected log pattern:
```
INFO  Workers: prefill=1, decode=1
INFO  Observed: num_req=150.00 isl=512.00 osl=128.00
INFO  Observed: ttft=45.00ms itl=8.50ms
INFO  Predicted: num_req=160.00 isl=520.00 osl=130.00
INFO  Prefill: 462.22(tok/s) / 5500.00(engine_cap) = 1(replicas)
INFO  Decode: 115.56(tok/s) / 750.00(engine_cap) = 1(replicas)
INFO  Target replicas: prefill=1, decode=1
```

When load increases, you'll see the planner scale up:

```
INFO  Workers: prefill=1, decode=1
INFO  Observed: num_req=1200.00 isl=2048.00 osl=256.00
INFO  Observed: ttft=180.00ms itl=18.50ms
INFO  Predicted: num_req=1350.00 isl=2100.00 osl=260.00
INFO  Prefill: 15750.00(tok/s) / 2800.00(engine_cap) = 6(replicas)
INFO  Decode: 1950.00(tok/s) / 500.00(engine_cap) = 4(replicas)
WARNING GPU budget exceeded (10 > 8), scaled to prefill=4, decode=4
INFO  Target replicas: prefill=4, decode=4
INFO  Scaling role prefill: 1 -> 4
INFO  Scaling role decode: 1 -> 4
```

Watch RBG replicas change in real-time:

```bash
kubectl get rbg sglang-pd-inference -w
```

### Step 5: Monitor via Prometheus (Optional)

If `PLANNER_PROMETHEUS_PORT` is enabled (default: 9091), query planner metrics:

```bash
# Port-forward planner metrics
kubectl port-forward svc/s-sglang-pd-inference-planner 9091:9091

# Query correction factors
curl -s localhost:9091/metrics | grep rbg_planner_p_correction_factor
curl -s localhost:9091/metrics | grep rbg_planner_predicted_num
```

## Configuration

All configuration via environment variables (or CLI args as overrides):

| Variable | Default | Description |
|----------|---------|-------------|
| `RBG_NAME` | (required) | Name of the RoleBasedGroup |
| `RBG_NAMESPACE` | `default` | Kubernetes namespace |
| `PREFILL_ROLE_NAME` | `prefill` | Prefill role name in the RBG |
| `DECODE_ROLE_NAME` | `decode` | Decode role name in the RBG |
| `PROMETHEUS_ENDPOINT` | `http://prometheus-kube-prometheus-prometheus.monitoring.svc.cluster.local:9090` | Prometheus URL |
| `METRIC_SOURCE` | `sglang` | Metric source: `sglang`, `vllm`, or `patio` |
| `MODEL_NAME` | (empty) | Model name for Prometheus label filtering |
| `ADJUSTMENT_INTERVAL` | `180` | Seconds between scaling decisions |
| `MAX_GPU_BUDGET` | `8` | Maximum total GPUs (prefill + decode) |
| `MIN_REPLICAS` | `1` | Minimum replicas per role |
| `PREFILL_ENGINE_NUM_GPU` | `1` | GPUs per prefill engine instance |
| `DECODE_ENGINE_NUM_GPU` | `1` | GPUs per decode engine instance |
| `TTFT_SLA` | `500.0` | Target TTFT in milliseconds |
| `ITL_SLA` | `50.0` | Target ITL in milliseconds |
| `LOAD_PREDICTOR` | `arima` | Predictor: `constant`, `arima`, or `prophet` |
| `LOAD_PREDICTION_WINDOW_SIZE` | `50` | Data points in predictor window |
| `NO_CORRECTION` | `false` | Disable SLA correction factors |
| `PROFILE_RESULTS_DIR` | `/etc/rbg-planner/profiling` | Profiling data mount path |
| `PLANNER_PROMETHEUS_PORT` | `0` | Planner metrics port (0=disabled) |
| `NO_OPERATION` | `false` | Dry-run mode (observe only, no scaling) |

## Profiling Data

The planner requires profiling data that maps load characteristics to hardware requirements. This data can be obtained through:

1. **inference-ext-cli** — Built-in SLA profiling pipeline (`inference-ext-cli profile`). Deploys temporary RBG instances, runs AIPerf benchmarks, and generates profiling JSON files.
2. **AIC (AI Center)** — Enterprise profiling service that produces compatible profiling data at scale.

The profiling data is mounted as a Kubernetes ConfigMap with two JSON files:

**prefill_raw_data.json** - Maps input sequence length to TTFT and throughput/GPU:
```json
{
  "prefill_isl": [128, 256, 512, 1024, 2048],
  "prefill_ttft": [0.008, 0.015, 0.030, 0.060, 0.120],
  "prefill_thpt_per_gpu": [6000, 5000, 4000, 3000, 2000]
}
```

**decode_raw_data.json** - Scatter points mapping (KV usage, context length) to (ITL, throughput/GPU):
```json
{
  "x_kv_usage": [0.02, 0.05, 0.10, 0.20, 0.35, 0.50, 0.70, 0.90],
  "y_context_length": [350, 350, 700, 700, 1400, 1400, 2800, 2800],
  "z_itl": [0.006, 0.008, 0.009, 0.012, 0.017, 0.022, 0.035, 0.055],
  "z_thpt_per_gpu": [1500, 1350, 1250, 1050, 750, 550, 400, 250],
  "max_kv_tokens": 32768
}
```

## Prometheus Metrics

When `PLANNER_PROMETHEUS_PORT` is set (e.g., `9091`), the planner exposes its own Prometheus metrics at `/metrics`. These metrics provide full observability into the planner's decision-making process.

### Worker Counts & GPU Usage

| Metric | Description |
|--------|-------------|
| `rbg_planner_num_prefill_workers` | Current prefill replica count |
| `rbg_planner_num_decode_workers` | Current decode replica count |
| `rbg_planner_gpu_hours_total` | Cumulative GPU hours used since planner start |

### Observed Metrics

Metrics observed from the inference endpoints via Prometheus in each adjustment interval:

| Metric | Description |
|--------|-------------|
| `rbg_planner_observed_ttft_ms` | Observed average Time to First Token (ms) |
| `rbg_planner_observed_itl_ms` | Observed average Inter-Token Latency (ms) |
| `rbg_planner_observed_request_rate` | Observed request rate (req/s) |
| `rbg_planner_observed_request_duration_seconds` | Observed average request duration (s) |
| `rbg_planner_observed_isl` | Observed average input sequence length (tokens) |
| `rbg_planner_observed_osl` | Observed average output sequence length (tokens) |

### Predicted Metrics

Load predictions and scaling decisions computed by the planner:

| Metric | Description |
|--------|-------------|
| `rbg_planner_predicted_request_rate` | Predicted request rate for next interval (req/s) |
| `rbg_planner_predicted_isl` | Predicted input sequence length (tokens) |
| `rbg_planner_predicted_osl` | Predicted output sequence length (tokens) |
| `rbg_planner_predicted_num_prefill` | Predicted number of prefill replicas needed |
| `rbg_planner_predicted_num_decode` | Predicted number of decode replicas needed |

### Correction Factors

Correction factors that calibrate profiling-based predictions against real-world performance:

| Metric | Description |
|--------|-------------|
| `rbg_planner_p_correction_factor` | Prefill correction factor (observed TTFT / expected TTFT) |
| `rbg_planner_d_correction_factor` | Decode correction factor (observed ITL / expected ITL) |

## Grafana Dashboard

A pre-built Grafana dashboard is provided at `deploy/grafana-planner-dashboard.yaml`. It visualizes all planner metrics in four sections:

1. **Worker Counts & GPU Usage** — Current prefill/decode worker counts (stat panels), cumulative GPU hours, and worker count history over time
2. **Observed Metrics** — TTFT & ITL latency (dual-axis), request rate & duration (dual-axis), ISL & OSL sequence lengths
3. **Predicted Metrics** — Predicted request rate, predicted ISL/OSL, and predicted replica counts (dashed lines to distinguish from observed)
4. **Correction Factors** — Prefill/decode correction factor gauges with threshold coloring (green < 1.2, orange < 1.5, red >= 1.5), and correction factor history over time

### Deploying the Dashboard

```bash
# Deploy the dashboard ConfigMap (Grafana auto-discovers it via the grafana_dashboard label)
kubectl apply -f deploy/grafana-planner-dashboard.yaml

# If using a different monitoring namespace, update the metadata.namespace field first
```

The dashboard includes a **Namespace** variable dropdown to filter metrics when running multiple planners across namespaces.

### Prometheus Scraping

To have Prometheus scrape the planner's metrics endpoint, add a scrape config or PodMonitor targeting the planner pod on the configured `PLANNER_PROMETHEUS_PORT` (default: `9091`):

```yaml
scrape_configs:
  - job_name: 'rbg-planner'
    static_configs:
      - targets: ['<planner-pod>:9091']
```
